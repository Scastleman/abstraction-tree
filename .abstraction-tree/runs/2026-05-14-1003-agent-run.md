## What Improved

`atree serve --open` now starts the local app server and opens the target URL in the default browser only when the user explicitly asks for it. Plain `atree serve` still only prints the URL.

Browser opening is cross-platform through a small utility that selects `cmd /c start`, `open`, or `xdg-open`. Failure to launch a browser is non-fatal and prints the URL for manual opening.

`atree init --with-app` and full-mode `atree scan` now guide users toward `atree serve --project ... --open`, so an external project setup naturally ends by inspecting the generated tree in the visual app.

Docs now explain the recommended setup flow and clarify that the app reads the target project's own `.abstraction-tree/` memory, not this repository's dogfooding memory unless this repository is the target.

## What Did Not Improve

The test suite does not launch a real browser. It verifies command selection, failure handling, CLI help parsing, and setup guidance without opening external applications.

## Risks

Platform browser launch commands are intentionally simple and dependency-free. Some Linux or remote/headless environments may lack `xdg-open`; the CLI now treats that as a warning rather than a server failure.

## Missing Context Discovered

Validation correctly fails before a memory refresh when new CLI files exist outside `.abstraction-tree/files.json`. The expected path is to run `atree scan`, then validate again.

## Tree Updates Needed

The repo abstraction memory was refreshed with `npm.cmd run atree:scan`, then validated and evaluated.

## Reusable Lesson

Adoption flows should end with the human seeing the generated abstraction tree, but browser launch must be opt-in so CI, SSH, and headless users stay in control.

## Recommended Next Loop

Add an optional post-serve readiness probe or app health message that confirms `/api/state` is reachable before printing browser guidance.
