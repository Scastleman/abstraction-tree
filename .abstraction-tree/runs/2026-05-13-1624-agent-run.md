## What Improved

The root project node now starts with inferred human subsystem nodes when evidence exists. For this repository, the first layer is Visual App / Explorer, Core Engine, CLI and Local API, Goal and Mission Automation, Memory / Validation / Logs, Docs and Examples, Packaging and Adapters, Tests and Quality Gates, and Project Indexes.

The previous domain, architecture, and code buckets remain under `project.indexes`, so context packs, validation, and agent lookup still have deterministic support views.

The tree builder now refuses to invent an app subsystem when UI evidence is absent and falls back to top-level repository areas when no strong subsystem patterns are available.

## What Did Not Improve

The subsystem inference is still deterministic and heuristic. It does not yet infer deep product concepts the way a reviewed LLM adapter could.

## Risks

Subsystem nodes can overlap on files because a file may belong to both a human responsibility and an index/architecture/module boundary. That is intentional for navigation, but agents should still narrow to modules or files before editing.

## Missing Context Discovered

The scanner still ignores `.abstraction-tree/` as source input, so the Memory / Validation / Logs subsystem represents the source code and docs that manage memory, not the generated memory files themselves.

## Tree Updates Needed

None after the scan. The refreshed tree includes the new human subsystem layer and `Project Indexes`.

## Reusable Lesson

Human-first trees should infer subsystem nodes from evidence and keep deterministic indexes as support views, rather than forcing every repository through fixed buckets like app, engine, or logs.

## Recommended Next Loop

Improve subsystem inference with more fixture projects so non-app repositories show natural first-layer nodes such as service, library, compiler, data pipeline, or research workflow.
