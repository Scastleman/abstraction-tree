## What Improved

`npm run lint` now checks scoped public documentation and current prompt surfaces for unsafe autonomy claims. It blocks unqualified claims and allows lines framed as explicit non-goals, historical terminology, compatibility language, or bounded non-claim wording.

The test coverage now proves that a direct claim in README text is reported, while "What This Is Not", historical "full self-improvement loop" wording, and "without claiming autonomous correctness" wording are allowed.

## What Did Not Improve

The rule is intentionally heuristic. It does not classify all possible autonomy claims, does not parse multi-paragraph nuance, and does not scan archived run reports or lessons.

## Mistakes / Risks

The first targeted test command used `node --test`, which failed with sandbox `spawn EPERM`. The repo's direct script-test import path and full `npm.cmd test` runner both passed.

The allowlist is line and current-heading based, so future docs with unusual structure may need a small rule adjustment.

## Missing Context Discovered

Node's built-in `--test` process-spawning mode is not reliable in this sandbox, while `node scripts/run-tests.mjs` remains the appropriate local test path.

## Tree Updates Needed

## Reusable Lesson

Treat public autonomy language as a lintable safety boundary, not only a documentation preference. Keep the guard scoped to current public surfaces so historical run memory stays readable.

## Recommended Next Loop

Document the `no-unsafe-autonomy-claim` lint rule in `docs/AGENT_PROTOCOL.md` so future agents know how to phrase autonomy non-goals before editing public docs.
