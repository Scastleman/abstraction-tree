## What Improved

External project initialization now creates only a neutral project-local workspace skeleton: `config.json`, `changes/`, and `context-packs/`. It no longer pre-creates feature runtime directories that could make the boundary between blank workspace and generated memory less clear.

The package smoke test now fails if an npm package includes root `.abstraction-tree/` dogfooding memory, verifies clean initialized workspace contents, and verifies that scan output in the installed smoke project is generated from that project rather than this repository.

The doctor command now detects obvious Abstraction Tree self-memory contamination in non-Abstraction-Tree projects and reports a clear warning with recovery guidance.

Docs now explicitly explain that this repository's root `.abstraction-tree/` folder is dogfooding memory only, not starter memory for users.

## What Did Not Improve

The contamination detector is intentionally heuristic. It catches obvious copied Abstraction Tree memory, but it is not a general provenance system for every possible stale workspace.

## Risks

The new doctor check relies on recognizable self-memory markers. If this repository's generated memory shape changes significantly, the marker list may need to be updated.

## Missing Context Discovered

The install smoke project can include package manager artifacts such as `package-lock.json` after package installation, so tests should check for project-local memory rather than an exact file list after scan.

## Tree Updates Needed

The repo abstraction memory was refreshed with `npm.cmd run atree:scan`, then validated and evaluated.

## Reusable Lesson

Dogfooding memory must be treated as project-owned runtime knowledge, not as a package template. Installation tests should assert the boundary directly.

## Recommended Next Loop

Add a small `atree doctor --repair-plan` style output that prints the exact cleanup commands for contaminated workspaces without deleting anything automatically.
