## What Improved

The CLI change-review path now has package-level tests for both bounded detail output and invalid limit input. The command action was extracted into `runChangeReviewCommand` with injectable IO, so future CLI option tests can avoid brittle subprocess spawning.

## What Did Not Improve

Generated scan-record retention was not fixed. Evaluation still warns that 144 older generated scan records are eligible for consolidation.

## Mistakes / Risks

The first test design used child-process spawning and failed under this sandbox. The final design covers the command action in-process, but it still does not test Commander argument wiring end to end.

Regenerating abstraction memory changed generated JSON broadly enough for `diff:summary:windows` to flag possible overreach, even though the source change is limited.

## Missing Context Discovered

Node child_process spawning can be unavailable in the local Codex Windows sandbox, so CLI tests that spawn `node packages/cli/dist/index.js` are less portable than injected command helpers.

## Tree Updates Needed

None after this loop. `npm run atree:scan` refreshed root memory and strict validation passed.

## Reusable Lesson

For CLI behavior coverage, prefer small action helpers with injected IO over child-process tests. Node child_process may be unavailable in Codex/Windows sandboxes, while in-process helpers still protect option validation and output shape.

Add a dry-run generated scan-record consolidation command that preserves semantic change records and retains the newest generated scan baseline.
