## What Improved

The README now tells future agents to use `atree changes review --project /path/to/project --summary` for a compact count-only report before deciding whether the full generated scan candidate list is needed.

Objective evidence from this checkout: the compact summary is 326 characters instead of 79,855 characters for the full report.

## What Did Not Improve

This loop did not change change-record retention behavior. Evaluation still reports generated scan buildup: 125 generated scan records, 124 older generated scan records eligible for consolidation, 24 semantic records, and 0 change-review issues.

## Mistakes / Risks

Refreshing abstraction memory added one generated scan record. That was necessary for strict validation after the README edit, but it slightly increases the known generated scan-record buildup.

## Missing Context Discovered

The full-loop doc already mentioned compact change-review summary usage, but the README CLI reference lagged behind the implemented command.

## Tree Updates Needed

## Reusable Lesson

When a compact diagnostic already exists, expose it in the README command reference; otherwise future agents may spend tokens on verbose JSON that is only needed after the summary indicates a reason to inspect details.

## Recommended Next Loop

Add a bounded `--limit <n>` option to `atree changes review` so agents can inspect a small sample of eligible generated scan records without printing the entire candidate list.
