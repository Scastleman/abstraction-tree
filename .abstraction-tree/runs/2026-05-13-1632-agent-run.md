Rebuild human subsystem branches so they decompose into responsibility slices and file leaves.

## Hypothesis

The tree is more useful for reliability and overreach restriction when first-layer human subsystems do not end as large aggregate nodes. Each subsystem should expose smaller responsibility boundaries and concrete files before an agent edits.

## What Improved

Human subsystem nodes now have children. Each subsystem decomposes into responsibility slices, and each slice has concrete file leaves that reference the same scanned files also available through Project Indexes.

For example, Visual App / Explorer now expands into App Shell and State, UI Components, Styling, and Visual Support. App Shell and State expands into file leaves such as `package.json`, `App.tsx`, `main.tsx`, and `nodeAccessors.ts`.

## What Did Not Improve

The slice classifier is still deterministic and rule-based. It uses visible path, package, file-role, script, doc, and test evidence, not deep semantic inference.

## Risks

The tree is larger: 436 nodes instead of 214. This is a good tradeoff for navigability, but context-pack scoring and app rendering should be watched as the repo grows.

Files can appear in both subsystem branches and Project Indexes. This is intentional because subsystem branches are human navigation views and Project Indexes are deterministic lookup views.

## Missing Context Discovered

The earlier subsystem-first tree was too shallow because it relied on `sourceFiles` ownership but did not create child nodes. The visual app only renders the `children` hierarchy, so ownership alone is not enough for human navigation.

## Tree Updates Needed

None after the scan. The refreshed tree contains the new subsystem responsibility layer and file leaves.

## Reusable Lesson

A human-facing abstraction tree should not stop at aggregate subsystem names. If a node can own many files, it should also explain its internal partition rule and provide navigable child nodes down to concrete file leaves.

## Recommended Next Loop

Add fixtures for different project types so the subsystem slice classifier can be tested against libraries, APIs, CLI tools, UI apps, and documentation-heavy repos instead of being tuned only on this repository.
